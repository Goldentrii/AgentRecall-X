Naming system v1 — the foundation

You asked specifically for a reasonable naming system, because retrieval quality is bottlenecked by name discoverability. Today AgentRecall stores under 5 inconsistent conventions (YYYY-MM-DD.md, NNNN-slug.md, free-form room slugs, etc.) — an agent asking "what did I learn about Cloudflare" has to grep against all of them.

Grammar

<scope>/<type>/[<topic>/]<temporal>--<slug>.md

scope    project | global
type     episodic | semantic | procedural | narrative | correction | insight
topic    optional sub-category (e.g. "deploy" for procedural skills)
temporal ISO date (YYYY-MM-DD), ISO datetime, or 4-digit NNNN ordinal
slug     kebab-case lowercase, ≤ 40 chars

Examples

prismma-gateway/episodic/2026-05-30--cloudflare-dns-fix.md
prismma-gateway/semantic/routing--cloudflare-zone-config.md
prismma-gateway/procedural/deploy--cloudflare-4step-pattern.md
prismma-gateway/narrative/0007--ssl-regression-fix.md
prismma-gateway/correction/2026-05-29--no-push-without-permission.md
global/insight/correction-first-memory-pattern.md

Why this grammar

  1. Type maps to the formal memory taxonomy — episodic/semantic/procedural are the canonical joints Squire (2004), Tulving (1972), ACT-R, and CoALA cut at. AgentRecall now carves at the right joints, not at "what folder did this end up in."
  2. Temporal prefix sorts naturally — `YYYY-MM-DD` is lex-monotonic; `NNNN` is numeric-monotonic; both order correctly with no custom comparator.
  3. Grep-friendly across layers — grep -r "cloudflare" ~/.agent-recall/projects/ finds every reference regardless of which layer wrote it.
  4. Compositional — an agent constructs paths from intent without "what room does this go in?" guessing.
  5. Backward compatible — existing files keep working. dashboard.json exposes a canonical view on top of legacy storage (we synthesize canonical entries from legacy filenames). No migration required.

Implementation

packages/core/src/naming.ts — 200 lines. Exports:

  • toSlug(input) — safe kebab-case, strips dots (no traversal), ≤40 chars.
  • canonicalPath(name) — build path from CanonicalName struct.
  • parseCanonicalName(path) — roundtrip, returns null on mismatch.
  • buildIndexEntry(path, headline) — produce a NamingIndexEntry (dashboard schema).
  • legacyToCanonicalType(path) — translator for in-place dashboard view.

Correction KPIs (V9) — the moat made measurable

The V9 finding: AgentRecall claims a learning loop but only measures retrieval. The only KPI that matters is recurrence after retrieval. We now track 3 events per correction:

retrieved
surfaced via check/recall
heeded
agent's next action honored it
recurred
same bug happened anyway

Schema

interface CorrectionRecord {
  // ...existing fields...
  retrieved_count?: number;
  heeded_count?: number;
  recurrence_count?: number;
  precision?: number;        // heeded / retrieved
  last_retrieved?: string;
  last_outcome?: string;
}

interface CorrectionOutcome {
  correction_id: string;
  project: string;
  kind: "retrieved" | "heeded" | "recurred";
  at: string;
  evidence?: string;
}

Aggregate KPI

getCorrectionKPIs(project) → {
  total, active, retrieved, heeded, recurred,
  precision: heeded / retrieved,
  noise_candidates: [...],  // precision < 0.3 with ≥3 retrievals → archive
  high_signal:     [...],   // precision ≥ 0.8 with ≥3 retrievals → promote faster
}
This converts the moat from a slogan into a number. If recurrence_count > 0 after promotion to awareness, the system failed and the insight needs reformulation — not re-display. Without this, "AgentRecall learns" is unfalsifiable.

FSRS-lite (V5) — memory decays adaptively

Ebbinghaus (1885) → SuperMemo SM-2 → FSRS-6 (Anki ≥23.10) all converge on: memory needs decay and reinforcement. AgentRecall today is write-only — confirmations only grow, salience only grows, nothing ever decays. By year 3-5 the corpus self-contradicts.

R = exp(-days_since_lastConfirmed / S)

initFsrs()               → { stability: 7, last_confirmed: now, confirmations: 1 }
score(state)             → { retrievability, stability, age_days, status }
reinforce(state)         → S grows 30%, lastConfirmed = now, confirmations++
penalize(state)          → S halves (kept lastConfirmed so age stays)

status buckets:
  R ≥ 0.85  →  "hot"
  R ≥ 0.6   →  "warm"
  R ≥ 0.3   →  "cool"
  R < 0.3   →  "archive_candidate"

This is the foundation. Wiring it into recall (so retrieval reinforces) and into session_end (so weekly scoring tags archive candidates) is the immediate next step — but the primitive lands today and nothing depends on it being wired before it can be used.

Procedural skills (V10) — the 5th memory layer

Today's biggest structural addition. Agents constantly re-derive the same multi-step procedures every session. palace/skills/ stores IF-THEN production rules with this schema:

interface SkillBody {
  when: string;              // The IF
  preconditions: string[];   // Must hold before applying
  steps: string[];           // Ordered execution
  postconditions: string[];  // Falsifiable success criteria
  pitfalls?: string[];       // Known failure modes
  evidence?: string[];       // journal refs / commit SHAs / correction ids
}

Reflection bundle (V2) — Park-style aggregation

The V2 finding: AgentRecall only promotes insights from corrections. Successful patterns never become semantic knowledge. Park et al. 2023 (Generative Agents) showed that periodic LLM-driven distillation is the move that makes memory compound.

session_end_reflect bundles last-N journals + active corrections + recent phase syntheses into a ready-to-use prompt the calling LLM reads, and outputs structured suggestions: distill skills, promote insights, archive noise.

Design choice: core does not call an LLM. The agent already in the conversation does the synthesis on its own turn and feeds results back via skill_write or session_end. Matches AgentRecall's "agent-as-author" stance.

Closing — memory is the foundation, the loop is the moat

What today's pass actually proves

  1. Naming was the bottleneck. Once the canonical grammar is in place, agents can ask "skills/deploy/cloudflare-*" without grepping. The index makes retrieval compositional.
  2. Procedural memory was the structural gap. Adding the 5th layer unblocks agents from re-deriving solved procedures every session — which is the daily cost the V6 vantage flagged.
  3. The learning loop needs a KPI. Until V9, "AgentRecall learns" was unfalsifiable. The recurrence counter makes it falsifiable. That's the only thing that turns the moat from claim to artifact.
  4. Memory must decay. FSRS-lite is the primitive. Wiring it into recall is one line in a follow-up.
  5. Agents must self-inspect. dashboard.json is the dual surface to the human dashboard.html — agent-first principle now lives in code, not just in CLAUDE.md.

What's missing is wiring, not foundations. The next pass connects FSRS to recall, syncs skills+corrections to Supabase, ships the dashboard UI cards (graph + correction timeline), and adds the V7 reducer when multi-agent contention becomes real.
